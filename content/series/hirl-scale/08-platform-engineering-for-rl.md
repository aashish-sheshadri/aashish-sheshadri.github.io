RL papers describe the agent. Conference talks show the learning curves. Blog posts cover the reward function design.

Almost nobody talks about the platform engineering required to make an RL system safe to operate in production. That's a problem, because in my experience the agent is roughly 20% of the engineering effort. The other 80% is the infrastructure that makes it trustworthy, debuggable, and survivable when things go wrong.

This post is the 80%.

## The Telemetry Pipeline: Your Agent Is Only as Good as Its Inputs

The Commander and Soldiers receive observations built from cluster metrics. If those metrics are wrong, delayed, or missing, the agents make decisions based on fiction. At the scale of 100k pods, this isn't hypothetical — metric pipelines fail in subtle ways all the time.

![Telemetry Pipeline](/images/08_telemetry_pipeline.svg)

### The Aggregation Service Is Load-Bearing Infrastructure

I treat the aggregation service with the same reliability standards as any payment-critical pipeline — because if the RL agents receive stale observations during a payment spike, the scaling decisions they make with those stale inputs can degrade the entire checkout flow.

The aggregation service should run with N+2 redundancy (two hot standbys), automatic leader election, and a strict freshness check: if any domain's metrics haven't been updated within 30 seconds, the affected domain's feature vector is flagged as stale and the agent falls back to HPA for that domain.

This "partial fallback" design matters. You never want the failure of one domain's telemetry to affect the RL decisions for other domains. Each domain should be independently monitored for metric freshness.

### Feature Normalization Is a First-Class Concern

Neural networks are sensitive to input scale. An unnormalized observation where CPU is in range [0, 100] and RPS is in range [0, 100000] will have wildly different gradient magnitudes per feature. In practice, this causes the policy to effectively ignore low-magnitude features.

I normalize all features to [0, 1] range using running statistics (exponential moving average of mean and std) computed by the aggregation service. The statistics update continuously on a slow timescale (τ = 1 hour) so they don't shift faster than the policy can track.

What happens when a feature's distribution shifts (e.g., after a major traffic growth event)? The normalization statistics shift slowly, causing a transient period where the feature is effectively mis-scaled. Monitor for this: if a Commander observation feature's raw value exceeds 2× the current normalizer's expected maximum for 3 consecutive ticks, fire an alert and have an engineer review the normalization config.

## The Action Executor: Where RL Meets Reality

Every action generated by a Soldier passes through the action executor before touching Kubernetes. The executor is not a thin wrapper — it's a safety system.

### Hard Constraint Layer

Before any action executes, the executor validates five categories of constraint: minimum replica floors, maximum replica ceilings, per-app scale velocity limits, budget cap against the Commander's allocation, and a cluster-wide node ceiling. These are hard rules. No RL action can violate them, even if the policy has degraded or is producing nonsense. They encode physical and operational constraints that the reward function cannot reliably enforce by itself.

One intentional design choice: Soldiers can slightly exceed their Commander budget in genuine emergencies — latency breaching SLO right now. But the overage is capped, not unlimited, to prevent a Soldier from perpetually ignoring the budget under the justification of "emergency." Where exactly to set that cap is a values question as much as an engineering one.

### Rollback Trigger

The executor continuously monitors post-action outcomes. If a scaling action is followed by a cluster-wide p99 latency increase of >2× within 5 minutes, the executor:

1. Freezes all pending RL actions
2. Issues scale-back commands to return apps to their pre-action replica counts
3. Re-enables HPA for the affected domains
4. Pages on-call with the action log, pre-action observation, and post-action metrics

The 5-minute window is chosen to be long enough to observe the actual outcome of a scaling action (pod startup time + warmup = 2–4 minutes typical) but short enough to bound the damage window if the action was bad.

### Rate Limiting

The executor rate-limits Kubernetes API calls to stay within API server throughput limits. A Soldier generating scale-up actions for all 500 apps in its domain simultaneously would produce 500 HPA patch calls in a single 10-second tick. Across all five Soldiers, that could mean up to 1,500 patches. The API server does not appreciate this.

Batch HPA patches per-domain and apply them over an 8-second window (leaving 2 seconds before the next Soldier tick). Within the domain, prioritize patches by at-risk score — the most-critical apps get their patches first.

## Observability: The Scaling Decision Ledger

When something goes wrong with an RL-based autoscaler, the first question is always "why did it do that?" Without per-decision logging, this question is unanswerable, and the team loses trust in the system permanently.

The **scaling decision ledger** is an append-only log of every Commander and Soldier decision, with full context:

```text
Commander decision record:
  timestamp
  observation_vector (full, before normalization)
  forecast_values (per domain, per horizon)
  action_taken (budget_pct, urgency, headroom per domain)
  actor_logits (raw network outputs)
  value_estimate (Critic output)
  actual_outcome (observed at t+60s)

Soldier decision record:
  timestamp
  domain_id
  observation_vector
  commander_directive_received
  action_taken (replica_delta per app)
  top_k_apps_at_decision_time (name, metrics, delta issued)
  actual_outcome (observed at t+10s)
```

These records stream to the observability stack (a dedicated Kafka topic → ClickHouse pipeline works well). Every record is indexed by timestamp, domain, and app. On-call engineers should be able to answer:

- "Why did Domain 3 get 38% cluster budget at 14:32?" → Pull Commander record at 14:32, inspect observation + action
- "Why did service X get scaled to 45 replicas?" → Pull Soldier record for Domain 3, find service X in top-K apps
- "Is the Commander using the forecast?" → Correlation query: forecast_value vs action_urgency over last 7 days

The ledger also feeds an automated reward hacking detector. Once a week, run a correlation analysis between the reward signal and every available cluster metric — including metrics not in the reward function. Any metric that's significantly more correlated with reward than with actual SLO outcomes is a candidate for being gamed.

## Human Override: The Feature That Builds Trust

Every engineer on-call should be able to override the system at any time. This is not a concession to distrust — it's a prerequisite for operational trust. Teams adopt systems they feel they can control.

Three levels of override:

**Domain budget pin:** An SRE can pin a domain's budget allocation for up to 4 hours. The Commander's allocation for that domain is fixed; the rest of the cluster is still managed by HiRL-Scale. Use case: planned maintenance or promotion when human judgment should override the learned policy.

**App replica pin:** An SRE can pin a specific app's replica count for up to 2 hours. The Soldier sees the pin and excludes the app from its scaling decisions. Use case: an app with a known bug that causes OOM at high replica counts.

**Domain fallback:** A domain can be completely handed back to HPA, indefinitely, with a single command. No other domains are affected. Use case: major incident investigation, new deployment that might interact badly with RL scaling.

All overrides are logged with username, timestamp, justification, and expected duration. Override frequency is a leading indicator of trust — if overrides are increasing week-over-week, something is generating decisions the team doesn't trust, and you need to find out what.

One thing I'm genuinely uncertain about: whether making overrides this easy creates the wrong long-term incentive. If engineers know they can always pin a domain, they have less pressure to file bugs when the RL behavior seems wrong. The system gets overridden rather than fixed. I've tried to counteract this by tracking override justifications and routing them back to the model improvement queue — but I don't know yet whether that feedback loop is tight enough in practice.

## Monitoring the RL System Itself

Beyond cluster metrics, monitor the health of the RL system as a system:

**Policy entropy.** The actor network outputs a probability distribution over actions. Entropy measures how "decisive" the policy is. Very high entropy = random policy, probably undertrained or distribution-shifted. Very low entropy = overconfident policy, possibly overfit. Alert if entropy crosses either threshold.

**Value function error.** The Critic predicts future cumulative reward. Track the mean absolute error between Critic predictions and actual observed returns. Consistent high error indicates the value function is out of distribution.

**Forecast vs actual accuracy.** Compare the TFT's p80 forecasts against actual observed RPS at the corresponding horizons. Calibration drift (p80 exceeded more than 25% of the time) triggers a forecaster refresh.

**Coupling reward trend.** If the Commander alignment bonus received by Soldiers is decreasing over time, the Commander and Soldiers are drifting out of sync. Early warning for cooperative breakdown.

**Reward component attribution.** Track each term in the Commander and Soldier reward functions separately. If the cost efficiency term is improving while SLO adherence is staying flat, that's suspicious — efficiency at the expense of SLO, potentially a reward hacking signal.

## The Incident That Changed How I Think About Safety

Early in shadow mode, I identified a scenario where the Commander's policy, if executed, would have made a seriously bad decision: a simulated peak traffic event caused the Commander to allocate 85% of cluster budget to Domain 1 (Payment Processing), leaving Domain 2 (Fraud Detection) with 8%. Domain 2's minimum floor would have kicked in, but the 7% headroom between "minimum floor" and "adequate capacity" was insufficient for the Fraud Detection domain's actual load at that time. A P1 domain in SLO breach with no mechanism for rapid recovery.

This caused me to revisit the minimum floor settings (raised them), add explicit domain capacity floor validation to the Commander's observation (the Commander now sees each domain's current load as a fraction of its minimum floor), and add a "floor proximity alert" in the Commander's reward: if any domain's actual utilization is within 15% of its minimum floor, the Commander pays an additional penalty.

The minimum floor is a hard constraint, but hard constraints alone don't prevent the Commander from navigating the cluster into a state where domains are barely above their floors with no headroom. The reward change ensures the Commander actively maintains headroom, not just floor compliance.

## Compliance and Audit Considerations

In financial services, any system that makes automated decisions affecting transaction processing carries regulatory weight. HiRL-Scale doesn't process transactions directly, but its scaling decisions determine whether transaction-processing services have adequate capacity — which makes it operationally load-bearing for payment availability.

**The decision ledger as audit trail.** The scaling decision ledger described above isn't just an operational debugging tool — it's an audit artifact. Regulators and internal audit teams need to understand why automated systems behaved the way they did during incidents. The ledger provides a complete chain: observation → forecast → action → outcome, for every Commander and Soldier decision. Retention policy: 7 years, consistent with financial record-keeping requirements.

**Model governance (SR 11-7 or equivalent).** Banking regulators expect model risk management practices (SR 11-7) for any model that influences business-critical operations. Even fintechs without bank charters often adopt these practices as a condition of bank partnership agreements. HiRL-Scale's RL policies and traffic forecaster fall under this umbrella. In practice, this means:
- Documented model development process (the training curriculum serves as this)
- Ongoing performance monitoring (the reward component attribution and calibration tracking)
- Independent validation before production deployment (shadow mode serves this role)
- Change management for model updates — any reward function change or retraining cycle requires review and approval before canary rollout

**Reward function changes as policy changes.** In a fintech context, modifying the reward function isn't just a hyperparameter update — it's a change to the system's values. Adjusting priority weights between Payment Processing and Fraud Detection has compliance implications. I treat reward function changes with the same change management rigor as changes to transaction processing logic: documented rationale, peer review, staged rollout, and rollback plan.

**Incident response integration.** During a payment processing incident, the RL system needs to be immediately controllable. The human override mechanisms described above aren't just operational convenience — they're a regulatory expectation. Automated systems affecting payment availability must have manual override capability with clear escalation paths.

## The Platform Contract

Here's what I'd put in writing with any team running this system:

- **Scaling decisions are explainable.** For any scaling action in the last 7 days, the on-call can see why it happened, in under 2 minutes.
- **Humans can always override.** Any engineer can pin, freeze, or fall back any domain, immediately, with a single command.
- **Rollback is automatic.** If RL actions cause measured latency increases, rollback happens within 5 minutes, automatically, without human intervention.
- **No surprise blast radius.** An RL failure in one domain does not cascade to other domains. Domains are independently monitored and can independently fall back.
- **Weekly review.** The on-call team reviews reward curves, override frequency, and scaling decision samples every week. If the review generates concerns, they are addressed before the next canary traffic increase.
- **Regulatory audit trail.** Every scaling decision affecting payment-critical domains is logged with full context, retained for 7 years, and queryable by compliance teams within 24 hours of request.
- **Change management for reward functions.** Any modification to the reward function or priority weights requires documented rationale, peer review, and staged rollout — equivalent to the change management process for transaction processing logic.

This contract isn't in the architecture doc. It's in the runbook, and it's what actually determines whether the team trusts the system enough to run it at 100% traffic.
